# UI 长度硬约束设计

## 背景
当前流程对 `UI / 按钮 / 标签 / 短词组` 的长度没有独立约束。模型在语义正确的前提下，容易给出明显长于中文的英文或印尼语短文案，导致多语言游戏实际接入时出现超框风险。

## 目标
- 只对 `UI / 按钮 / 标签 / 短词组` 生效。
- 保持“自然可懂优先，长度第二”。
- 让模型在审核时主动压缩短文案长度。
- 对超预算的 UI 文案增加程序级硬约束，避免直接放行。

## 范围
纳入：
- UI 文案
- 按钮
- 标签
- 短词组

不纳入：
- 完整句子
- 邮件正文
- 叙事对白
- 含复杂占位符或富文本的长说明

## 设计
### 1. UI 长度预算判定
新增 `ui_length_checker`，输入原文、译文、UI 标记和目标语言，输出：
- 是否为长度约束候选
- 中文长度
- 目标语言长度
- 预算上限
- 是否超预算

候选条件：
- 已被 `ui_detector` 判为 UI
- 原文长度落在短文案区间
- 译文不含复杂长句结构

### 2. 硬约束规则
对候选条目，如果目标语言长度超出预算上限，则生成新问题类型：
- `ui_length_overflow`

该问题默认进入人工/AI 复审，不自动放行。

### 3. Prompt 注入
主审 prompt 增加一段 UI 长度规则说明：
- 仅对 UI / 按钮 / 标签 / 短词组生效
- 在自然可懂的前提下尽量贴近中文长度
- 英文可略长，但不要明显膨胀
- 如果某行附带 `LEN` 元数据，必须尽量在预算内改写

同时对需要长度控制的行增加元数据：
- `UI:yes`
- `LEN: source={n}, target={m}, budget<={k}`

### 4. 二次回流
AI 回填后重新跑长度检查。
如果仍然超预算，则继续保留 `ui_length_overflow`，进入待确认列表或二次批次，不直接吞掉。

## 数据流
1. 机审完成 UI 判定
2. 运行 UI 长度预算检查
3. 把 `ui_length_overflow` 放进 AI 主审队列
4. Prompt 注入长度规则和行级预算信息
5. AI 回填
6. 再次运行 UI 长度预算检查
7. 输出最终结果和报告

## 影响文件
- `utils/ai_checker.py`
- `process_language.py`
- `tests/test_ai_review_protocol.py`
- `tests/test_process_language.py` 或新增 UI 长度测试文件
- 新增 `utils/ui_length_checker.py`

## 风险与处理
- 预算过紧会误伤自然表达
  - 用分段预算，不做“英文必须不长于中文”的死规则
- 误把完整句子当短 UI
  - 只对 UI 且短文本启用
- 模型为了压长度生成缩写腔
  - prompt 明确“自然可懂优先”

## 成功标准
- Prompt 中可见明确的 UI 长度约束
- 超框高风险短文案会被标成 `ui_length_overflow`
- AI 回填后，超预算 UI 文案数量下降
- 不影响完整句子类文案
