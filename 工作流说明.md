# 本地化质检工作流说明

## 1. 当前可用入口

这个项目现在有 3 个主要入口：

- `workspace_runner.py`
  - 推荐入口
  - 适合按工作区目录批量处理项目
  - 支持 `machine / prepare / merge`
- `cli.py`
  - 适合单个语言表直接跑
  - 支持严格 AI 批次生成和回填
- `gui.py`
  - 适合人工操作
  - 仍可用，但当前主流程建议优先走 `workspace_runner.py` 或 `cli.py`

核心处理逻辑在：

- `process_language.py`
- `utils/ai_checker.py`

## 2. 当前完整链路

### 阶段 A：预检与机审

程序会先读取语言表和术语表，然后完成：

- 源语言 / 目标语言检测
- 术语表加载
- 变量与占位符检查
- BBCode / 换行检查
- 术语命中检查
- 拼音残留专名检查
- 中文残留检查
- 句式一致性检查
- UI 文本识别
- UI 短文案长度预算检查

高置信问题会自动修复。

### 阶段 B：严格 AI 批次生成

`prepare` 阶段会在输出目录下生成：

- `ai_review\batch_N.txt`
- `ai_review\batch_N.json`
- `ai_review\batch_N_response.txt`

如果有术语二次复查，还会生成：

- `ai_review\batch_recheck_N.txt`
- `ai_review\batch_recheck_N.json`
- `ai_review\batch_recheck_N_response.txt`

其中：

- `txt` 是给 AI 审核看的批次内容
- `json` 是严格 manifest，绑定每个 ID 和输入指纹
- `response.txt` 是默认预填好的回复模板，初始全是 `KEEP`

### 阶段 C：AI 审核填写

每条回复只能是两种格式之一：

- `ID | KEEP`
- `ID | FIX | corrected translation`

要求：

- 必须覆盖该批次全部 ID
- 不能缺行
- 不能多行
- 不能输出解释、标题、总结、代码块
- 如果行里带有 `LEN:mode=...,source=...,target=...,budget<=...`，说明这是一条短文本长度约束
- `mode=hard` 表示紧凑 UI 文案，预算更强
- `mode=soft` 表示普通短文本，优先尽量紧凑，但不能为了压长度破坏自然度

### 阶段 D：严格 merge 回填

`merge` 阶段不会再从 prompt 文本里猜 ID，而是只按 manifest 合并。

merge 会检查：

- 批次是否全量覆盖
- 回复格式是否正确
- 当前输入是否还是 prepare 时的同一份文件
- 行指纹是否一致

只要不一致，就直接拒绝 merge。

## 3. 常用命令

### 3.1 单个项目目录

例如处理：

- `C:\Users\Administrator\Desktop\本地化处理\二合`

先生成严格批次：

```powershell
python cli.py --input "C:\Users\Administrator\Desktop\本地化处理\二合\二合语言表.xlsx" --term-base "C:\Users\Administrator\Desktop\本地化处理\二合\二合术语.xlsx" --lang en --output-dir "C:\Users\Administrator\Desktop\本地化处理\二合\output_en" --auto-fix --agent prepare --batch-size 80 --ai-scope all --strict-review
```

批次回填完后合并：

```powershell
python cli.py --input "C:\Users\Administrator\Desktop\本地化处理\二合\二合语言表.xlsx" --term-base "C:\Users\Administrator\Desktop\本地化处理\二合\二合术语.xlsx" --lang en --output-dir "C:\Users\Administrator\Desktop\本地化处理\二合\output_en" --auto-fix --agent merge --strict-review
```

### 3.2 工作区批量处理

例如处理：

- `C:\Users\Administrator\Desktop\本地化处理`

先按工作区生成批次：

```powershell
python workspace_runner.py --workspace "C:\Users\Administrator\Desktop\本地化处理" --project 土拨鼠 --lang auto --mode prepare --auto-fix --batch-size 80 --ai-scope all
```

批次回填后再 merge：

```powershell
python workspace_runner.py --workspace "C:\Users\Administrator\Desktop\本地化处理" --project 土拨鼠 --lang en --mode merge --auto-fix --strict-review
```

## 4. 输出文件

每次运行都会输出：

- `result_{lang}.xlsx`
- `report_{lang}.xlsx`
- `ai_review\`
- 一个按时间戳归档的结果目录

其中：

- `result.xlsx`
  - `完整结果`
  - `需确认`
  - 可选 `术语行筛选`
- `report.xlsx`
  - `总览`
  - `错误模式`
  - `学习笔记`
  - `详细记录`

## 5. 当前工作流解决了什么问题

相对于旧版，这套链路主要解决了这些问题：

- 不再从 prompt 文本反解析 ID，避免输入输出词条错配
- 默认给每个批次生成全量 `KEEP` 模板，避免少回、漏回
- 严格校验输入指纹，避免 prepare 和 merge 之间换表
- 主审之外再加术语二次复查，降低术语漏改
- 支持按目录自动识别语言表和术语表
- 增加拼音残留检查，降低 `Hongshangu / Jushizhen / Xigu...` 这类错误翻译漏网
- 增加短文本长度预算检查：中文原文 `<= 10` 先入池，再按紧凑 UI / 普通短文本 / 专名豁免分层处理

## 7. 注意事项

- `prepare` 和 `merge` 之间不要替换输入语言表
- 如果 response 文件被改乱格式，严格 merge 会直接失败
- 术语表里如果把泛词定义得过硬，仍然会出现术语误报，需要继续调术语表本身
- 小批次首跑优先稳定，建议 `80` 到 `100`
- 短文本长度检查不是“全部 10 字以内都硬压”，而是先入池再分层
- `ui_length_overflow` 主要用于紧凑 UI 文案
- `short_text_length_watch` 是普通短文本的软提示，不代表必须机械缩写
- 如果项目故意保留音译专名，建议显式放进术语表或白名单，否则可能被拼音残留规则拦下
